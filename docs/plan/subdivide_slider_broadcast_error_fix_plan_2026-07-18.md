# subdivide スライダー操作時の broadcast error 改善計画

作成日: 2026-07-18  
対象: `src/grafix/core/effects/subdivide.py`

## 状況

`sketch/readme/top_movie.py` でパラメータスライダーを連続操作すると、`subdivide`
の出力書き込み時に次の例外が発生する。

```text
ValueError: could not broadcast input array from shape (33,3) into shape (65,3)
```

描画ランタイムは直前の正常フレームへ退避できているが、操作中に scene evaluation
が失敗するため、この入力を正しく処理できるようにする。

## 調査結果

### 根本原因

現在の `subdivide` は、出力配列を一度だけ確保するために次の二つを別々に行っている。

1. `_subdivided_vertex_count()` が出力頂点数を事前見積もりする。
2. `_subdivide_core()` が float32 の中点を実際に生成し、最短線分長を毎回判定する。

事前見積もりは最初の最短距離へ解析的に `0.25` を掛け続ける。一方、実生成側は
float32 で丸められた中点から距離を再計算する。そのため、大きく平行移動した座標や
停止閾値付近では、見積もりが 6 段・65 頂点、実生成が 5 段・33 頂点になることがある。

呼び出し側は「見積もりと実生成は常に同数」と仮定し、33 頂点を 65 頂点分の
slice へ代入しているため、broadcast error になる。

### 再現確認

- 最小入力:
  - float32 の 2 点 `[[0, 0, 0], [0.32000002, 0, 0]]`
  - `subdivisions=10`
  - 見積もり 65 頂点、実生成 33 頂点
- `top_movie.py` 相当:
  - `polyhedron(center=(255.8, 0, 0))`
  - `fill()` の後に `subdivide(subdivisions=6)`
  - 同じ 65 対 33 の不一致を確認
- 現在進行中のパフォーマンス変更では `subdivide.py` は変更されていない。
  この不具合は、出力の事前確保を導入した既存実装の問題である。

## 修正方針

事前計算値は厳密な出力数ではなく、安全な**確保容量の上限**として扱う。実際の
書き込み位置・offset・診断値については、`_subdivide_core()` の実出力を唯一の
真実源にする。

解析式を float32 の実生成へ無理に一致させる方法は採用しない。丸め誤差は絶対座標にも
依存するため、同種の境界値を完全には排除できないためである。

## 実装項目

- [x] 境界値の最小再現テストを追加し、修正前に同じ例外が発生することを確認する。
- [x] `_subdivide_core()` から実出力と実際に適用した細分回数を取得できるようにする。
- [x] 事前見積もり頂点数を出力バッファの capacity としてのみ利用する。
- [x] 各ポリラインの書き込み範囲と `offsets_out` を実出力頂点数から構築する。
- [x] 見積もりと実出力が異なる稀な場合だけ、未使用領域を保持しない大きさへ出力を縮める。
- [x] minimum segment length の diagnostic を実際の適用回数から生成する。
- [x] 複数ポリラインでも順序・全頂点・offset が崩れない回帰テストを追加する。
- [x] `top_movie.py` 相当の effect chain を通す統合回帰テストを追加する。
- [x] vertex cap、通常の subdivide、diagnostic の既存テストを実行する。
- [x] 対象 lint・型検査と全 pytest を実行する。
- [x] 通常経路のベンチマークを取り、修正による有意な性能退行がないことを確認する。

## 受け入れ条件

- スライダーで問題の座標・細分回数を通過しても `RealizeError` が発生しない。
- 最小再現入力の出力 shape は `(33, 3)`、offset は `[0, 33]` になる。
- 複数ポリラインで `offsets[-1] == len(coords)` が常に成立する。
- diagnostic の effective 値は見積もりの 6 ではなく実適用回数の 5 になる。
- `MAX_TOTAL_VERTICES` を超えず、末尾ポリラインも欠落しない。
- 通常入力では余分な再生成や二重評価を追加せず、事前確保の性能上の利点を維持する。

## 対象外

- `MIN_SEG_LEN` や `MAX_SUBDIVISIONS` の仕様変更
- float32 から別 dtype への全面移行
- scene evaluation 失敗時のランタイム表示・ログ方針の変更

## 実施結果

- float32 境界値: `(65, 3)` の確保容量に対する実出力 `(33, 3)` を正常に格納
- offset: 単一線 `[0, 33]`、境界線と通常線の組み合わせ `[0, 33, 98]`
- diagnostic: 実適用回数 `5` を報告
- `top_movie.py` の保存パラメータ相当:
  - density `260.51`、`261.0`、`261.49` の連続範囲ですべて正常完了
  - 各出力 `1,009,512` 頂点、`offsets[-1] == len(coords)`
- 対象テスト: 20 passed
- 全テスト: 1607 passed
- Ruff: 対象3ファイルで成功
- mypy: `subdivide.py` で成功
- 通常経路ベンチマーク:
  - 1,000頂点・1線: -4.68%
  - 50,000頂点・1線: -0.47%
  - 64頂点・256線: +0.04%
  - 2頂点・5,000線: +3.33%
  - 通常的な単一線・中規模複数線は同等で、極端な多短線ケースも許容基準の +5% 以内
